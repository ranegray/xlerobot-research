# v0.2 Developer Velocity and Fake Runtime

Window: May 1-May 15

Theme: make the repo easy to build, test, and extend.

This release exists so coding agents can safely produce useful PR-sized changes.

## Main Goal

A coding agent can safely make changes to tests, docs, configs, schemas, fake-runtime support, and analysis scripts without destabilizing the hardware path.

## Deliverables

| area | deliverable | repo artifact |
| --- | --- | --- |
| Dev environment | one-command setup or clear setup docs | `Makefile`, `docs/development/setup.md` |
| CI | lint, unit tests, static launch contracts, package build, fake launch smoke | `.github/workflows/ci.yml`, `scripts/check_launch_contracts.py`, `docs/development/launch_inventory.md` |
| Issue structure | backlog grouped by runtime, harness, benchmark, data, policy, agent, docs | `docs/project/backlog.md`, `.github/ISSUE_TEMPLATE/backlog_item.yml` |
| Guard tests | unit tests for valid and invalid trajectory cases | `tests/test_trajectory_guard.py`, `tests/fixtures/trajectories/` |
| Fake runtime docs | launch, command, observe, and record fake one-arm runs | `docs/simulation/fake_hardware.md` |
| Agent contribution rules | allowed/protected surfaces for coding agents | `AGENTS.md`, `.github/pull_request_template.md` |
| Data guardrails | run-card shape for fake and real evidence | `examples/run_cards/fake_one_arm_smoke.yaml`, `scripts/validate_run_card.py` |
| Event schemas | stable guard event JSON shape | `xle_hardware/harness_events.py`, `tests/test_harness_events.py` |

## Agent-Sized Jobs

- generate more guard and harness fixtures in `tests/fixtures/`
- add schemas for run cards and replication bundles
- expand fake launch smoke into command-and-observe smoke
- improve package inventory checks
- write benchmark protocol fixtures
- add analysis scripts that read bags but do not write hardware commands

## Done When

The repo has a passing local `make ci-local` path and a ROS CI path that builds packages, launches fake one-arm bring-up, observes fake joint state, publishes a valid guarded trajectory fixture, and sees a `command_accepted` harness event.
